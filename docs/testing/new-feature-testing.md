# 游戏新功能测试指南

## 目标

新功能测试的目标是确认功能在预期场景下能稳定运行，并且在异常或边界情况下不会破坏已有游戏流程。测试记录需要让其他成员能复现：测试了什么、测试前做了什么准备、运行了哪个场景、观察到了哪些日志或表现。

## 需要测试的内容

### 功能主流程

- 玩家或系统触发新功能后，核心逻辑是否按设计顺序执行。
- 节点状态是否能正确切换，例如待机、启动、运行中、结束、重置。
- 输入、碰撞、信号、计时器、动画、物理效果等关键事件是否在正确时机发生。
- 功能结束后，节点是否回到可再次使用的状态。

### 边界和异常情况

- 连续快速触发功能时，是否出现重复执行、状态卡死或资源残留。
- 功能在无目标、目标消失、节点被禁用、场景切换等情况下是否安全退出。
- 参数取最小值、最大值或空值时，逻辑是否仍符合预期。
- 功能失败后是否能恢复，例如重新进入待机状态或重新允许输入。

### 与其他系统的交互

- 新功能是否影响玩家控制、镜头、HUD、暂停菜单、关卡加载或全局状态。
- 信号连接是否只触发一次，是否存在重复连接导致的重复回调。
- 物理节点是否正确处理碰撞层、碰撞遮罩、刚体状态和区域检测。
- 场景切换、重新开始关卡、暂停/恢复后，新功能是否仍然可用。

### 性能和可维护性

- 功能运行时是否持续打印大量无意义日志。
- `_process`、`_physics_process` 或信号回调中是否存在明显的重复创建对象、重复查找节点或无限累积数据。
- 调试日志是否能通过开关关闭，避免影响正式试玩。

## 前置工作

### 1. 明确测试点

在开始测试前，先把功能拆成可观察的测试点：

- 触发条件：什么输入、碰撞、信号或状态会启动功能。
- 关键状态：功能运行过程中会经过哪些状态。
- 预期结果：每个状态应该带来什么画面、物理、数值或信号变化。
- 退出条件：功能什么时候结束，结束后节点应该恢复成什么状态。

建议把这些测试点写成简短清单，避免只凭试玩感觉判断是否通过。

### 2. 添加临时日志输出

如果需要测试某个节点的运行逻辑是否符合预期，应先在关键路径加日志。日志需要覆盖：

- 进入和退出关键函数，例如 `_ready`、`_physics_process`、输入处理函数、碰撞回调、信号回调。
- 状态切换，例如 `idle -> active`、`active -> finished`。
- 关键参数，例如位置、速度、目标节点、计时器剩余时间、当前状态名。
- 异常分支，例如目标不存在、条件不满足、提前返回。

日志格式建议包含节点名、功能名和状态，方便在 Godot 输出面板中筛选：

```gdscript
print("[AnchorTest] state=", _state, " position=", global_position, " velocity=", _velocity)
```

临时日志应在功能验证完成后清理，或改成可通过调试开关控制的日志，避免污染正式输出。

### 3. 搭建测试场景

为新功能准备一个最小测试场景，场景只保留测试所需的节点和依赖，避免完整关卡中太多变量干扰判断。

测试场景应覆盖：

- 正常路径：按预期触发功能并完成完整流程。
- 失败路径：触发条件不完整或目标不存在时，功能能安全退出。
- 边界路径：快速重复触发、最大距离、最短时间、切换场景、暂停恢复等。

场景命名建议放在 `debug/` 下，例如：

```text
debug/FeatureNameRegression.tscn
debug/feature_name_regression.gd
```

如果测试脚本需要自动触发输入或移动节点，应把测试步骤写在脚本中，并在关键阶段打印日志。

### 4. 准备可复现的初始状态

运行测试前需要确保场景初始条件固定：

- 相关节点的位置、旋转、速度、碰撞层和碰撞遮罩已配置。
- 必要的 Autoload、输入映射和资源引用可用。
- 测试目标节点已加入正确的 group，信号连接已完成。
- 随机数、计时器或物理参数不会导致结果每次都不同。

如果某个功能依赖玩家输入，测试场景可以提供脚本化触发，减少人工操作误差。

## 测试流程

### 1. 运行结构或脚本检查

如果功能修改了脚本结构，先运行已有的项目结构测试：

```bash
python -m unittest tests/scaffold/test_project_structure.py
```

这一步用于尽早发现脚本、场景路径、必要方法或导出变量缺失等问题。

### 2. 运行 Godot 测试场景

打开 Godot 后直接运行对应的 `debug/` 测试场景。观察内容包括：

- 输出日志顺序是否符合预期。
- 节点状态是否按测试清单切换。
- 画面、物理运动、碰撞和信号触发是否正确。
- 测试结束后是否没有报错、警告或状态残留。

如果需要无界面验证，可使用 Godot headless 运行项目或指定场景，至少确认脚本解析和场景加载没有错误。

### 3. 对照日志判断结果

测试时不要只看画面表现，也要对照日志确认内部逻辑。

例如测试一个节点的运行逻辑：

1. 给节点关键函数和状态切换加日志。
2. 搭建一个覆盖正常路径和异常路径的测试场景。
3. 运行测试场景。
4. 在输出面板中检查日志顺序、状态值、位置/速度等关键参数。
5. 如果日志和画面都符合预期，再清理临时日志或改成调试开关。

### 4. 回归完整游戏流程

单独测试场景通过后，还需要回到真实游戏场景中试玩，确认新功能不会破坏主流程。至少检查：

- 从主场景进入关卡后功能可用。
- 暂停、继续、重开、切换场景后功能仍然正常。
- HUD、音效、动画、碰撞和玩家控制没有被意外影响。

## 完成标准

一个新功能可以认为通过测试，需要满足：

- 有明确测试场景或可复现的测试步骤。
- 正常路径、失败路径和至少一个边界路径已经验证。
- 关键日志能证明内部状态变化符合预期。
- Godot 运行时没有脚本错误、场景加载错误或明显警告。
- 临时调试内容已清理，或保留为可控的调试开关。
- 在真实游戏流程中试玩后没有破坏已有功能。
